# 融合渲染架构

## 定义

CMP 融合渲染架构是一种将 JetBrains Compose Multiplatform 的声明式 UI 渲染管线与 OHOS 平台的 RenderService 原生渲染能力相融合的方案，通过 SkPicture 录制 → OH_Drawing 命令转换 → ContentModifier 聚合 → RenderNode 挂载的完整数据流，实现高性能原生渲染。

## 详解

### 为什么选择融合渲染

1. **平台差异**：OHOS 使用独特的 RenderNode + OH_Drawing 渲染模型，与 Android/iOS 截然不同
2. **性能要求**：需保持 60fps 流畅渲染，同时支持复杂 UI 动画
3. **统一数据流**：将 Compose 绘制 → SkPicture → OH_Drawing → ContentModifier → RenderNode 形成完整链路

### 三阶段绑定模型

融合渲染中的内容绑定分为三个阶段：

| 阶段 | 触发时机 | 关键组件 | 代码位置 |
|------|---------|---------|---------|
| **编译时绑定** | 编译期 | Compose 编译器 → 组合树 | Compose Compiler |
| **ArkTS 运行时绑定** | 组件创建时 | `NodeContainer` + `CanvasNodeController` | `Compose.ets:151` |
| **Compose 运行时绑定** | 首次 surface 就绪后 | `ComposeSceneMediator.setContent()` | `ComposeSceneMediator.ohos.kt:130` |

### 四层数据流

```
ArkUI 层 (ArkTS)
  └→ Compose 层 (Kotlin)
       └→ OHRender 层 (C++)
            └→ OH_Drawing API 层
                 └→ RenderService 层
```

### 帧回调机制

融合渲染使用基于 `NodeContainer` + `RenderFrameCallback` 的帧回调：

- **触发事件**：willDraw、触摸事件、尺寸变化
- **回调链**：`notifyRedraw()` → `postFrameCallback()` → `CanvasRenderNode.draw()` → `OHRenderNode.nodeDraw()` → Compose 渲染
- **VSync 驱动**：通过 `clipToFrame` 属性触发下一帧 VSync

### 核心组件

| 组件 | 语言 | 职责 |
|------|------|------|
| [[SkPictureRecorder]] | C++ | 指令录制和脏区计算 |
| [[SkCanvas]] | C++ | 绘制命令转换和脏区跟踪 |
| [[OHRenderNode]] | C++ | RenderNode 管理和 Picture 回放 |
| `ComposeNodeController` | ArkTS | NodeController 实现，管理节点树 |
| `ComposeSceneMediator` | Kotlin | Compose 场景中介 |

### Compose 层数据结构

```
LayoutNode (布局节点)
  ↓ NodeCoordinator 管理
OwnedLayer (绘制层接口)
  ↓ skiko 平台实现
RenderNodeLayer (PictureRecorder 录制)
  ↓ 生成
SkPicture (绘制指令容器)
```

- **LayoutNode**：Compose UI 框架中布局树的核心节点
- **OwnedLayer**：绘制层的抽象接口（`drawLayer`、`invalidate`、`resize`、`destroy`）
- **RenderNodeLayer**：skiko 平台实现，使用 PictureRecorder 录制为 SkPicture，维护 Picture 缓存

## 关键要点

- 融合渲染通过 **统一数据流** 将 Compose 与 OHOS RenderService 连接
- 三阶段绑定确保组件生命周期的正确管理
- 帧回调由 NodeContainer 驱动，而非 XComponent
- RenderNodeLayer 的 Picture 缓存机制避免重复录制
