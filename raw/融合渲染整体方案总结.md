# CMP-OHRender融合渲染整体方案总结

## 📋 文档概述

本文档总结了Compose Multiplatform (CMP) 对接OpenHarmony OHRender的融合渲染整体方案，涵盖指令录制与回放机制、GraphicsLayer属性应用、以及ComposeUI与ArkUI混排原理。

---

## 🎯 核心方案概述

### 整体架构（四层架构）

融合渲染方案采用分层架构，从上到下依次为：

```
┌─────────────────────────────────────────────────────┐
│  1. Compose UI层 (Kotlin)                           │
│     - LayoutNode树：Compose布局节点                  │
│     - RenderNodeLayer：图层管理与Picture缓存         │
│     - GraphicsLayerScope：属性管理                   │
└──────────────────┬──────────────────────────────────┘
                   │ JNI调用
┌──────────────────▼──────────────────────────────────┐
│  2. Skiko适配层 (Kotlin/Native)                     │
│     - PictureRecorder：Picture录制器                 │
│     - Canvas：绘制接口封装                           │
│     - Picture：绘制命令容器                          │
└──────────────────┬──────────────────────────────────┘
                   │ Native API
┌──────────────────▼──────────────────────────────────┐
│  3. OHRender渲染层 (C++)                            │
│     - SkPictureRecorder：命令录制                    │
│     - SkCanvas：命令转换                             │
│     - SkOHPicture：Picture容器                       │
│     - OHRenderNode：渲染节点树                       │
│     - 模式决策：Picture模式 vs Node模式              │
└──────────────────┬──────────────────────────────────┘
                   │ System API
┌──────────────────▼──────────────────────────────────┐
│  4. HarmonyOS原生层 (C API)                         │
│     - OH_Drawing_Canvas：绘制API                    │
│     - OH_Drawing_RecordCmd：命令容器                │
│     - OH_ArkUI_RenderNodeUtils：节点管理            │
│     - ContentModifier：命令聚合                     │
│     - RenderService：硬件加速渲染                   │
└─────────────────────────────────────────────────────┘
```

## 🔄 核心流程：从LayoutNode到RenderService

### 简化示例场景

以下代码展示了一个典型的嵌套GraphicsLayer场景：

```kotlin
@Composable
fun ExampleUI() {
    Box(
        modifier = Modifier
            .size(200.dp)
            .graphicsLayer {
                clip = true
                shape = RoundedCornerShape(16.dp)
            }
            .background(Color.Blue)
    ) {
        Box(
            modifier = Modifier
                .size(100.dp)
                .graphicsLayer {
                    scaleX = 1.5f
                    scaleY = 1.5f
                    alpha = 0.8f
                }
                .background(Color.Red, CircleShape)
        )
    }
}
```

### 转化流程

#### 步骤1：LayoutNode树 → RenderNodeLayer树

```
LayoutNode(外层Box)
  └── NodeCoordinator
      └── OwnedLayer实现 → RenderNodeLayer A
          ├── GraphicsLayer属性：clip=true, shape=RoundRect(16dp)
          └── drawBlock: {
                LayoutNode(内层Box)
                  └── NodeCoordinator
                      └── RenderNodeLayer B
                          ├── GraphicsLayer属性：scale=1.5, alpha=0.8
                          └── drawBlock: { 绘制红色圆形 }
              }
```

**关键点**：
- 每个带有`graphicsLayer`修饰符的Composable都会创建一个`RenderNodeLayer`
- RenderNodeLayer负责管理图层属性和Picture缓存

#### 步骤2：Picture录制（嵌套录制机制）

```
RenderNodeLayer A.drawLayer()
  ├── [检查缓存] picture == null?
  │
  └── [开始录制]
      ├── PictureRecorder.beginRecording()
      │   ├── OH_Drawing_RecordCmdUtilsBeginRecording() → OH_Drawing_Canvas
      │   ├── OHRenderNode.CreateNormalNode()
      │   └── new SkCanvas(ohCanvas, node)
      │
      ├── [执行绘制]
      │   ├── drawBackground(Blue)
      │   ├── clipRoundRect(16dp)
      │   │
      │   └── RenderNodeLayer B.drawLayer() [嵌套Layer]
      │       ├── PictureRecorder.beginRecording() [SubCanvas]
      │       ├── scale(1.5, 1.5)
      │       ├── saveLayer(alpha=0.8)
      │       ├── drawCircle(Red)
      │       ├── finishRecording() → Picture B
      │       └── Canvas.drawPicture(Picture B) [立即回放]
      │
      └── [完成录制]
          ├── Canvas.getFinishDrawBounds() → 计算脏区
          ├── OHRenderNode.setRealFrame(脏区)
          ├── OH_Drawing_RecordCmdUtilsFinishRecording() → RecordCmd
          └── 创建SkOHPicture(cmd, node, 脏区)
```

**关键点**：
- **嵌套录制**：子Layer的Picture在父Layer录制过程中立即回放
- **脏区跟踪**：SkCanvas在每次绘制操作时实时更新`fDrawBounds`
- **命令转换**：Skia API → OH_Drawing API

#### 步骤3：Picture回放与模式决策

```
Picture A.playback(canvas)
  │
  ├── [获取父节点] canvas.getOHNode()
  │
  ├── [模式决策] ──────────────────────┐
  │                                   │
  │  条件判断：                        │
  │  • 无父节点？           → Picture  │
  │  • 在SaveLayer中？      → Picture  │
  │  • 无限制绘制？         → Picture  │
  │  • 之前在Node模式？     → Node     │
  │  • 绘制区域相交？       → Node     │
  │  • 其他               → Picture  │
  │                                   │
  ├─<─────────────────────────────────┘
  │
  ├── [Picture模式] 命令聚合
  │   ├── pictureDraw(canvas)
  │   ├── OH_Drawing_CanvasDrawRecordCmdNesting(canvas, recordCmd)
  │   └── 命令直接执行在父Canvas上
  │
  └── [Node模式] 独立节点
      ├── generateNewNode() [获取或克隆节点]
      ├── setParent(canvas_node)
      ├── updateFatherMatrix(matrix)
      ├── appendChild(node)
      └── 后续在帧回调时：
          └── nodeDraw(canvas)
              └── OH_Drawing_CanvasDrawRecordCmdNesting(canvas, recordCmd)
```

**模式对比**：

| 特性 | Picture模式 | Node模式 |
|:---|:---|:---|
| **渲染方式** | 命令聚合到父节点 | 独立RenderNode |
| **脏区管理** | 父节点统一管理 | 独立脏区 |
| **变换支持** | 继承父节点变换 | 独立变换矩阵 |
| **内存占用** | 低（无额外节点） | 高（节点对象开销） |
| **适用场景** | 静态/简单内容 | 动态/复杂/频繁变化 |
| **状态可逆** | 可转为Node | 不可逆（一旦分裂） |

**本例结果**：
- **外层Box（Layer A）**：Node模式（根节点）→ 创建OHRenderNode A
- **内层Box（Layer B）**：Picture模式（区域相交）→ 命令聚合到OHRenderNode A
- **最终**：仅1个OHRenderNode，包含外层+内层所有绘制命令

#### 步骤4：HarmonyOS渲染

```
OHRenderNode A
  ├── OH_Drawing_RecordCmd (聚合的绘制命令)
  │   ├── drawBackground(Blue)
  │   ├── clipRoundRect(16dp)
  │   ├── scale(1.5, 1.5)
  │   ├── saveLayer(alpha=0.8)
  │   └── drawCircle(Red)
  │
  └── [通过ContentModifier挂载]
      └── RenderService
          └── 硬件加速渲染
              └── 屏幕显示
```

## ⚙️ GraphicsLayer属性应用机制

### 属性分类

GraphicsLayerScope提供的属性可分为四类：

#### 1. 几何变换属性

| 属性 | 默认值 | 说明 | 实现方式 |
|:---|:---|:---|:---|
| `scaleX/Y` | 1.0f | 缩放 | 矩阵变换 |
| `rotationX/Y/Z` | 0f | 旋转（度） | 矩阵变换 + 透视 |
| `translationX/Y` | 0f | 平移（像素） | 矩阵变换 |
| `transformOrigin` | Center | 变换原点 | 矩阵计算中心点 |
| `cameraDistance` | 8.0f | 3D透视距离 | 透视矩阵元素 |

**矩阵计算流程**（RenderNodeLayer.updateMatrix()）：

```kotlin
matrix.reset()
matrix.translate(-pivotX, -pivotY)              // 1. 平移到原点
matrix *= rotateZ + rotateY + rotateX + scale   // 2. 旋转和缩放
if (rotationX != 0 || rotationY != 0) {
    matrix *= perspectiveMatrix(cameraDistance) // 3. 透视变换
}
matrix *= translate(pivot + translation)        // 4. 平移回去
```

**应用方式**：
- **Node模式**：`OH_ArkUI_RenderNodeUtils_SetTransform(node, matrix[16])`
- **Picture模式**：`OH_Drawing_CanvasConcatMatrix(canvas, matrix)`

#### 2. 裁剪属性

| 属性 | 默认值 | 说明 |
|:---|:---|:---|
| `clip` | false | 是否启用裁剪 |
| `shape` | RectangleShape | 裁剪形状 |

**支持的形状**：
- `RectangleShape`：矩形裁剪
- `RoundedCornerShape`：圆角矩形裁剪
- `GenericShape`：任意路径裁剪

**应用方式**：
- **Node模式**：`OH_ArkUI_RenderNodeUtils_SetClip(node, clipOption)`
- **Picture模式**：`OH_Drawing_CanvasClipRect/Path/RoundRect(canvas, ...)`

#### 3. 视觉效果属性

| 属性 | 默认值 | 说明 | 实现方式 |
|:---|:---|:---|:---|
| `alpha` | 1.0f | 透明度 | saveLayer() + Paint.alpha |
| `shadowElevation` | 0f | 阴影高度 | ShadowUtils.drawShadow() |
| `renderEffect` | null | 渲染特效（模糊等） | ImageFilter |

**Alpha实现策略**（依赖compositingStrategy）：

```kotlin
// 策略1：Auto（默认）
if (alpha < 1) {
    canvas.saveLayer(bounds, Paint().apply { alpha = this@Layer.alpha })
}

// 策略2：ModulateAlpha（高性能）
canvas.alphaMultiplier = alpha

// 策略3：Offscreen（强制离屏）
canvas.saveLayer(bounds, Paint())
```

#### 4. 合成策略属性

| 属性 | 说明 | 性能 |
|:---|:---|:---|
| `CompositingStrategy.Auto` | 自动决策（alpha<1时创建图层） | ★★☆ |
| `CompositingStrategy.Offscreen` | 强制离屏缓冲区 | ★★★ |
| `CompositingStrategy.ModulateAlpha` | 直接调制alpha（不保证正确合成） | ★☆☆ |

### 属性应用完整流程

```
用户代码设置属性
  └── Modifier.graphicsLayer { scaleX=1.5f; clip=true; ... }
      │
      ▼
GraphicsLayerScope (ReusableGraphicsLayerScope)
  └── mutatedFields位标记跟踪变化
      │
      ▼
RenderNodeLayer.updateLayerProperties(scope)
  ├── 复制属性值到RenderNodeLayer
  │
  ├── [如果变换属性变化]
  │   └── updateMatrix() → 计算4x4变换矩阵
  │
  └── invalidate() → 清除Picture缓存
      │
      ▼
下次drawLayer()时重新录制
  ├── performDrawLayer(canvas)
  │   ├── 应用阴影：drawShadow(canvas)
  │   ├── 应用裁剪：canvas.clipRect/Path/RoundRect()
  │   ├── 应用透明度：canvas.saveLayer() 或 alphaMultiplier
  │   └── 执行绘制：drawBlock(canvas)
  │
  └── 回放时应用变换
      │
      ├── [Picture模式]
      │   └── OH_Drawing_CanvasConcatMatrix(canvas, matrix)
      │   └── OH_Drawing_CanvasClipRoundRect(canvas, ...)
      │
      └── [Node模式]
          └── NodeStatusModify批量操作：
              ├── OH_ArkUI_RenderNodeUtils_SetTransform(node, matrix)
              ├── OH_ArkUI_RenderNodeUtils_SetClip(node, clipOption)
              └── OH_ArkUI_RenderNodeUtils_SetSize/Position(node, ...)
```

## 🚀 关键性能优化点

### 1. Picture缓存机制（RenderNodeLayer层）

**原理**：
```kotlin
override fun drawLayer(canvas: Canvas) {
    if (picture == null) {
        // 缓存未命中，重新录制
        val pictureCanvas = pictureRecorder.beginRecording(bounds)
        performDrawLayer(pictureCanvas, bounds)
        picture = pictureRecorder.finishRecordingAsPicture()
    }
    // 使用缓存
    canvas.drawPicture(picture)
}

override fun invalidate() {
    picture?.close()
    picture = null  // 清除缓存
}
```

**优势**：
- 避免重复录制相同内容
- 降低CPU开销
- 适用于静态或不常变化的内容

**失效时机**：
- 图层属性变化（`updateLayerProperties`）
- 图层大小变化（`resize`）
- 绘制内容变化（`drawBlock`变化时需手动`invalidate`）

### 2. 脏区管理（SkCanvas层）

**原理**：
```cpp
// 每次绘制操作时更新脏区
void SkCanvas::drawRect(const SkRect& rect, const SkPaint& paint) {
    // 执行OH_Drawing绘制
    OH_Drawing_CanvasDrawRect(fOHCanvas, rect);
    
    // 更新脏区边界
    updateDrawBounds(rect);  // fDrawBounds = fDrawBounds.join(rect)
}

// 录制结束时获取脚区
std::optional<SkRect> SkCanvas::getFinishDrawBounds() {
    return fDrawBounds;  // 返回合并后的边界框
}
```

**优势**：
- 只更新变化的区域，减少GPU渲染负担
- 通过`setRealFrame()`精确设置RenderNode尺寸
- 支持扩展不缩小策略，减少节点重建

### 3. 命令聚合（Picture模式）

**原理**：
- 多个子Layer的绘制命令聚合到同一个OHRenderNode
- 减少RenderNode数量，降低内存占用
- 适合静态内容的批量绘制

**示例**：
```
[独立Node模式]     [命令聚合Picture模式]
RenderNode A       RenderNode A
  ├── RecordCmd A    ├── RecordCmd A
  └── child:         │   ├── drawBackground
RenderNode B         │   ├── drawChild1
  └── RecordCmd B    │   ├── drawChild2
                     │   └── drawChild3
                     └── (无子节点)
```

**优势**：
- 减少节点对象开销（每个节点约1-2KB）
- 减少RenderService遍历开销
- 提升批量绘制效率

### 4. 节点复用（SkOHPicture层）

**原理**：
```cpp
class SkOHPicture {
    std::shared_ptr<OHRenderNode> fOriginNode;        // 原始节点
    mutable std::list<...> fCacheCloneNodes;          // 克隆节点缓存
    mutable std::list<...> fUnusedCloneNodes;         // 未使用节点池
};

std::shared_ptr<OHRenderNode> SkOHPicture::generateNewNode() {
    // 优先级1：从复用池获取
    if (!fUnusedCloneNodes.empty()) {
        return fUnusedCloneNodes.pop_front();
    }
    
    // 优先级2：使用原始节点
    if (fOriginNode && !fOriginNode->isInUse()) {
        return fOriginNode;
    }
    
    // 优先级3：克隆新节点
    return fOriginNode->makeClone();
}
```

**适用场景**：
- Crossfade动画（新旧内容同时显示）
- LazyList复用（同一模板多个实例）
- 并发回放

**优势**：
- 降低内存分配次数
- 避免频繁创建/销毁节点
- 提升动画性能

### 5. 批量属性同步（NodeStatusModify层）

**原理**：
```cpp
class NodeStatusModify {
    std::vector<NodeModifyInfo> fNodeOpInfos;  // 操作队列
    
    void pushTo(NodeOp op, OHRenderNode* node, data) {
        fNodeOpInfos.push_back({op, node, data});
    }
    
    void doModify() {
        for (auto& info : fNodeOpInfos) {
            switch (info.fOp) {
                case TRANSFORM:
                    OH_ArkUI_RenderNodeUtils_SetTransform(...);
                    break;
                case SIZE:
                    OH_ArkUI_RenderNodeUtils_SetSize(...);
                    break;
                // ...
            }
        }
        fNodeOpInfos.clear();
    }
};
```

**优势**：
- 减少JNI调用次数（从N次减少到1次）
- 批量操作提升效率
- 降低跨语言调用开销

## 🔑 关键技术决策

### 1. 为什么节点分裂后无法聚合？

**原因**：
- 一旦进入Node模式（`fDrawInPicture = false`），会创建独立的RenderNode
- 状态被锁定（`fCanPlaybackInPicture = false`）
- 避免频繁模式切换导致的性能抖动
- 子节点依赖关系无法轻易回退

**设计意图**：
- 保持状态稳定性
- 避免节点树结构破坏
- 减少模式切换带来的复杂性

### 2. 为什么需要嵌套录制？

**原因**：
- Compose UI层存在嵌套的`graphicsLayer`
- 每个Layer需要独立管理属性和Picture
- 子Layer的Picture需要在父Layer录制过程中回放

**机制**：
- SubCanvas机制：父Canvas创建子Canvas进行嵌套录制
- 立即回放：子Picture录制完成后立即在父Canvas上回放
- 数据流：Canvas链 → Picture树 → Node树

### 3. 为什么同时支持Picture和Node两种模式？

**原因**：
- **Picture模式**：适合静态内容，命令聚合，低内存占用
- **Node模式**：适合动态内容，独立脏区，精细化更新

**决策依据**：
- 内容复杂度
- 更新频率
- 绘制区域相交情况
- 父子关系
